## 问题分析

我出的场景是游戏玩家管理场景下，在一个玩家 ID 数组中找最大值的问题。题意比较简单，逻辑也不复杂。但是在 Round 1 自然语言描述时，我没有详细说明不定式的限制，应当要求玩家数组不能为空，且长度恒为 `10`：

```java
/*@ public invariant players != NULL && players.size() == 10;
 @*/
```

并说明抛出异常的名称为 `EmptyArrayException`：

```java
/*@ public exceptional_behavior
  @ assignable \nothing;
  @ signals (EmptyArrayException e) players.size() == 0;
@*/
```

Round 2 的同学在写 JML 时为我补上了，非常感谢。  

之后的自然语言和 JML 编写也没有问题。

## 个人感悟

### 对 JML 的感悟

JML 的可读性确实比较差。或许是因为其过于注重正确性而忽略了指导我们如何完成方法，而 JML 所描述的期望效果通过方法名就能猜个八九不离十。  

但是，JML 描述的各种规约对于开发是很有帮助的。有时候通过阅读 JML 可以发现自己漏掉的若干情况。  

总之，看起来 JML 更适合让 AI 阅读。  

### 对研讨课的感悟

这次研讨课的小游戏非常有趣，充分体现了一个无歧义的描述在团队协作中有多么重要的地位。  

另外让我体会到了写 JML 是一件非常痛苦的事情，即使十分小心还是会因为表述问题造成歧义，然后像滚雪球一样越来越偏。
